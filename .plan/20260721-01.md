# relay层安全修复与功能增强移植计划

## 执行规则

- 每完成一个具体步骤，必须把对应 checkbox 从 `[ ]` 更新为 `[x]`。
- 每完成一个 Phase，必须重新读取本文件，并提示下一步 Phase 的计划内容。
- 核心约束：绝对安全（不引入破坏性变更）、解耦计费（规则7禁止引入付费/计费/额度/倍率）、禁止音频/视频/任务（规则8）。
- 排除项：#2 param_override_error.go（已存在相同）、#4 reasonmap（已存在相同）、#6 usage.go（功能无差异仅文件组织）、#11 image_handler（无差异）、#12 rerank_handler（无差异）、#13 responses_handler音频路由（规则8排除）、Audio相关（规则8排除）。

## Current State

- relay层缺少URL脱敏日志功能（SanitizeURLForLog），API密钥等敏感信息可能泄露到日志
- 请求验证缺少MaxTokens边界检查，超大MaxTokens值可能导致预消费配额计算溢出
- GetAndValidOpenAIImageRequest使用c.MultipartForm()而非common.ParseMultipartFormReusable，缺少N验证和stream参数
- usage后处理函数内联在relay-openai.go中，文件组织可优化
- 缺少relayconvert统一转换框架、gemini/relay_responses.go、openai/responses_via_chat.go、advancedcustom渠道

## Target State

- 所有日志中的URL均经过脱敏处理，敏感query参数被mask
- 所有请求验证均包含MaxTokens/MaxOutputTokens边界检查
- GetAndValidOpenAIImageRequest使用正确的multipart解析方式
- usage后处理函数提取到独立文件
- relayconvert框架、gemini responses、openai responses_via_chat、advancedcustom渠道均已移植（排除计费/音频/任务依赖）
