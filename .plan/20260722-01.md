## Current State（问题清单）

按优先级分组，每项含定位与影响。

### A. 正确性 / 安全（P0）

- **C1 渠道表单 settings 往返闭包隐患**：`EditChannelModal.jsx:466–490` `handleChannelOtherSettingsChange` 在 480 行直接读渲染闭包中的 `inputs.settings`（非函数式），`handleChannelSettingsChange:461` 同样直接读 `channelSettings`；同一渲染周期内连续切换多个 settings 开关时，后一次读到旧值，先前修改在提交时被丢弃。`settings`（复数）blob 在提交 1578 行会带上该风险；`setting`（单数）blob 因提交 1567–1575 用显式白名单重建，污染不会持久化。与用户报告问题同族。
- **C2 规则 5 违规——上行 DTO 显式零值被吞**：`dto/openai_request.go:246–249` `StreamOptions.IncludeUsage/IncludeObfuscation` 为非指针 `bool`+`omitempty`；`dto/claude.go:203` `DisableParallelToolUse`；`dto/embedding.go:11–18` `Seed/TopK/NumPredict/NumCtx` 非指针 `int`+`omitempty`。客户端显式传 false/0 时被静默丢弃，未透传上游。
- **C3 CORS 非法组合**：`middleware/cors.go:10–12` 同时 `AllowAllOrigins=true` 且 `AllowCredentials=true`，违反 CORS 规范，跨域携带凭证被浏览器拒绝。
- **C4 SiderBar 可见性非响应式**：`components/layout/SiderBar.jsx:60–94` 把 `localStorage.getItem('enable_data_export')` 放入 `useMemo` 依赖，localStorage 非响应值，运行中变更不会刷新侧边栏。
- **C5 SettingsMonitoring 闭包 setState**：`pages/Setting/Operation/SettingsMonitoring.jsx:142–248` onChange 用 `setInputs({...inputs, k:v})` 非函数式，批量更新可能丢字段导致静默保存失败。
- **C6 upstream update 选择不存在列**：`controller/channel_upstream_update.go:50` 的 select 字段含 `"tag"`，但 `model.Channel` 无 `Tag`/`tag` 列（原版残留）。
- **C7 渠道 BaseURL 缺 SSRF 校验**：`controller/channel.go:437–495 validateChannel`（被 AddChannel/UpdateChannel 调用）未对管理员填写的 `channel.BaseURL` 调用已有的 `common.ValidateURLWithFetchSetting`，存在管理员级 SSRF/内网探测面。

### B. 性能 / 稳定性（P1）

- **P1 SQLite 连接池 + WAL**：`model/main.go:109–110` 默认 `SQL_MAX_OPEN_CONNS=1000`，与 SQLite 单写锁模型冲突，易 `SQLITE_BUSY`；`common/database.go` DSN 未启用 WAL。
- **P2 渠道选择热路径每请求 DB**：`model/channel_cache.go:210–218 getAbilityStatus` 每次查库，`filterDisabledModels` 在每个 relay 请求、每个候选渠道上同步读 SQLite，绕过内存缓存。
- **P3 日志分页双全扫**：`model/log.go:287–291`、`373–384` 每页 `COUNT(*)`+`SELECT` 两次扫描；用户侧在 `.Count()` 前的 `Limit(10000)` 对 count 无效（GORM 忽略）。
- **P4 user.Insert 多余往返**：`model/user.go:330–370` Create 后又按 username 查一次、再按 id 查一次并整体 Update，单次注册 4 次 DB 往返。
